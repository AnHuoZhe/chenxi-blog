# 晨汐博客 —— 需求描述

## 项目定位

个人技术博客。记录AI学习过程、展示项目作品。服务于求职（杭州AI岗位）和知识输出。

## 分类体系

博客内容分三个区：

### 细胞区（独立区）

代码的原子单位。变量、函数、数据类型、花括号等最基础的概念。
不参与任何层级嵌套，独立成区。
写作风格：生活化比喻，最少的话讲清原理和用途。

### 器官区（按用途分类）

有明确功能边界的模块。按用途分五大类：

- 前端：React、HTML/CSS、TailwindCSS
- 后端：FastAPI、Python标准库
- 部署：Git、GitHub Actions
- AI：LLM API、Function Calling、Agent Loop
- 数据：SQLite、JSON处理

每个器官页面包含器官说明正文。文中出现的组织专有名词自动链接到组织wiki页。

### 组织 Wiki

各器官涉及的基础概念（HTTP、JSON、环境变量、DOM、Virtual DOM等）。
不独立成章，淡化处理。从任意文章中的专有名词链接跳入，wiki风格展示。
每篇组织wiki：是什么、生活化比喻、原理和用途（一句话概括）。

### 个体区（独立区）

完整的项目。death-replay-agent、pareto-learner、晨汐博客本身等。
到达个体层不再挂在器官下面，独立展示。
每篇为项目复盘：架构决策、踩坑记录、学到了什么。

## 页面结构

### 首页

- 头像 + 名字（晨汐）
- 一句个人简介
- 顶部导航栏：首页 | 细胞 | 器官 | 个体
- 最近更新：展示最近5篇文章列表（跨所有区，按 frontmatter 中的 date 字段倒序排列）

### 细胞区列表页

- 按主题列出所有细胞级文章
- 每篇显示标题和一句话描述
- 按日期倒序排列

### 细胞文章页

- 文章正文（markdown渲染）
- 文中出现的组织术语和器官名自动链接（新标签页打开）
- 面包屑：首页 > 细胞 > 文章标题

### 器官区首页

- 按用途大类分组展示（前端/后端/部署/AI/数据）
- 每个大类下展开器官列表
- 每个器官显示名称和一句话描述

### 器官详情页

- 器官说明正文
- 文中出现的组织术语自动链接到组织wiki页（新标签页打开）
- 面包屑：首页 > 器官 > 器官名

### 组织 Wiki 页

- 简单干净的wiki风格
- 是什么 / 比喻 / 一句话原理和用途
- 面包屑：首页 > 组织wiki > 组织名

### 个体区列表页

- 所有项目列表
- 每个项目显示名称、一句话描述、状态
- 状态取值：施工中 / 已完成 / 归档
- 按日期倒序排列

### 个体项目页

- 项目复盘正文
- 文中出现的组织术语和器官名自动链接（新标签页打开）
- 面包屑：首页 > 个体 > 项目名

### 404 页面

- 极简页面：一句"这里什么都没有" + 返回首页链接

## 导航设计

- 所有页面顶部导航栏：首页 | 细胞 | 器官 | 个体
- 所有内页有面包屑导航，显示当前位置和回退路径
- 面包屑独立，不表达从属关系。组织wiki只写 `首页 > 组织wiki > 组织名`，器官只写 `首页 > 器官 > 器官名`
- 当前页面导航项高亮

## 写作风格（贯穿所有内容）

- 生活化比喻，最少的话讲清原理和用途
- 目的不是让人精通，是让人"有这个概念的印象，能看懂AI的输出"
- 细胞和组织特别强调此风格

## 技术栈

- 构建方式：Eleventy（11ty）静态网站生成器，基于Node.js
- 样式：手写CSS或CSS框架（如TailwindCSS），由Codex决定
- 部署：GitHub Pages
- 不需要后端、不需要数据库

构建命令：

```
npm run dev     # 本地开发预览，启动本地服务器，文件改动自动刷新
npm run build   # 构建，读取content/生成静态HTML到dist/
npm run deploy  # 部署，将dist/推送到GitHub Pages
```

## Markdown 文件元数据（Frontmatter）

每篇文章文件头部使用 YAML frontmatter 声明元数据：

```yaml
---
title: 变量是什么
description: 变量就是给数据贴个名字标签，要用的时候喊名字就行
date: 2026-07-10
---
```

字段说明：
- `title`：文章标题，必填。列表页和面包屑使用此字段。
- `description`：一句话描述，必填。列表页展示。
- `date`：写作日期，必填，格式 YYYY-MM-DD。用于排序。


## 自动链接机制

所有文章（细胞、器官、个体）启用自动链接。构建脚本的处理逻辑：

1. 扫描 `content/组织wiki/` 下所有 `.md` 文件，提取文件名（去掉 `.md`）作为**组织术语表**
2. 扫描 `content/器官/` 下所有 `.md` 文件，提取文件名作为**器官术语表**
3. 渲染任意文章时，扫描正文中是否出现两个术语表中的词
4. 匹配到的组织术语转为链接：`<a href="/组织wiki/[术语].html" target="_blank">术语</a>`
5. 匹配到的器官名转为链接：`<a href="/器官/[分类]/[器官名].html" target="_blank">器官名</a>`
6. 新标签页打开，不影响读者在原文章中的阅读位置

注意事项：
- 避免过度匹配——短术语（如"API"）可能误伤。对术语做长度过滤（如最少3个字符），或只匹配完整词边界。
- 组织和器官文件名不应重名，如有冲突人工改名解决。
- 术语匹配优先用最长匹配，避免"JSON"被"JSON处理"部分截断。
